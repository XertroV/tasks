Commands Reference

The Backlogs CLI provides a comprehensive set of commands for managing hierarchical project backlogs. All commands are available via the backlog, bl, or tasks entry points.

Commands are organized into core task operations, workflow shortcuts, reporting tools, and project management utilities. The table below lists all primary commands with their current deployment status.

Core Commands

These are the essential commands for day-to-day backlog management. Required commands are available on all platforms; optional commands provide additional functionality; deprecated commands will be removed in a future release.

Command Description Status
bl list List all tasks with filters required
bl grab Auto-claim next available task required
bl tree Show full task hierarchy optional
bl cycle Complete current + grab next required
bl done Mark task as complete required
bl show Display detailed task info optional
bl blocked Mark task as blocked optional
bl why Explain dependency readiness optional
bl search Full-text search across tasks deprecated

Workflow Shortcuts

Workflow shortcuts combine multiple operations into single commands. The most common pattern is the claim-work-complete loop, which can be executed manually or automated with bl cycle.

Example
# The standard workflow loop
backlog grab            # pick up next task
backlog show            # read the details
# ... do the work ...
backlog done            # complete it
backlog grab            # next one

# Or just use cycle to combine done + grab:
backlog cycle

For multi-agent environments, the grab command includes built-in claim locking and stale-claim detection. See the bl grab documentation for details on the scheduling algorithm and conflict resolution.

Information
All commands support --json output for programmatic consumption. Agent tooling commands (session, unclaim-stale) are designed specifically for automated workflows.

Pause and Hand Off

When work is blocked or needs to transfer between agents, use the dedicated commands:

Output
$ backlog blocked --reason "waiting on API key"
# Task P1.M2.E1.T042 marked as BLOCKED
# Reason: waiting on API key

$ backlog handoff --to agent-2 --notes "impl done, needs tests"
# Task P1.M2.E1.T042 transferred to agent-2
Service Advisory
Use bl handoff --notes to include context for the receiving agent. Good handoff notes reduce ramp-up time and prevent duplicate work.
